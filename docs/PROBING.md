# Morphological probing

A model-agnostic benchmark that measures whether a language model has learned
Turkish morphology — and the instrument that turns the tokenizer A/B into a
claim. Minimal-pair acceptability (BLiMP/CLAMS-style): each item is a
grammatical sentence and a minimally-ungrammatical counterpart; a model is
"correct" on an item when it assigns higher probability to the grammatical
member. Random baseline = 50%.

## The benchmark

`benchmarks/probe.jsonl` (version-controlled, deterministic) — 240 minimal
pairs, generated by `probe_dataset.py` using the morphological **generator**
as gold:

| category | tests | example (good / *bad) |
|---|---|---|
| `agreement` | subject–verb person | `ben geldim` / `*ben geldin` |
| `possessive` | possessor–suffix person | `benim evim` / `*benim evin` |
| `harmony` | vowel-harmony / allomorphy | `evde` / `*evda` |

The "bad" forms swap in a wrong agreement/possessive suffix or a
harmony-violating vowel, and are verified against the analyzer so they are
genuinely ill-formed (not an accidental real word).

Regenerate: `python probe_dataset.py` (deterministic).

## Running it

```bash
# stdlib baseline (no morphology -> ~chance, validates the benchmark)
python probe_eval.py --scorer char_unigram --corpus data/corpus.train.txt

# any Hugging Face causal LM (sanity baseline now, our own model later)
python probe_eval.py --scorer hf --hf-model <name-or-dir>
```

A scorer maps text → log-probability; `evaluate()` counts grammatical-preferred
pairs. A detailed report (`reports/probe_*.md` + `.json`) is written every run.

**Baseline check:** the char-unigram model scores **~43% overall** (agreement
50%, possessive 25%, harmony 55%) — at/below chance, confirming the pairs
require real morphological knowledge rather than surface character statistics.
A morphologically competent LM should score well above 50%.

## Role in the tokenizer A/B

Train two small LMs (Unigram-32k vs morpheme-BPE), wrap each as a scorer
(`--scorer hf` template), and compare probing accuracy alongside validation
loss. That comparison is the evidence for whether morphology-aware
tokenization actually helps the model — the project's thesis.
